### 3.1 推理业务分级矩阵

Gary的核心观点：集约化Serving丧失灵活性 → 需要分级部署策略。

```
                   延迟敏感度
                  低 ←──────→ 高
         ┌─────────────────────────┐
    高   │  视频生成(Seedance)     │  豆包问答                │
   并    │  特性：分钟级长任务      │  特性：<500ms首Token     │
   发    │  High throughput        │  High concurrency        │
   量    │                         │                          │
         │  ─────────────────────  │  ──────────────────────  │
         │  Agent(Coze/ArkClaw)    │  Trae代码生成            │
         │  特性：7×24持久连接     │  特性：<200ms首Token     │
         │  Steady state           │  Streaming + 长上下文    │
    低   │                         │                          │
         └─────────────────────────┘
```

| 推理业务 | 芯片分级建议 | 部署策略 | 关键优化点 |
|---------|------------|---------|-----------|
| **豆包C端** 120万亿Token/日 | 🔴 昇腾950PR主力 + 高通ASIC扩容 | 大规模推理集群，弹性伸缩 | KV Cache优化、Continuous Batching、量化(INT8/INT4) |
| **Trae代码** 低延迟流式 | 🔴 昇腾950PR低延迟链路 | 独立推理池，避免与其他混部干扰 | 投机解码(Speculative Decoding)、长上下文KV管理 |
| **视频生成** 分钟级任务 | 🟡 昇腾950PR + 寒武纪590 | 专用视频推理集群 | 批量调度、Diffusion加速、显存管理 |

## 五、基础设施层 TA：集约化Infra + 异构管理

### 5.1 集约化Infra的TA方案

```
目标：解决"集约化vs灵活性"的结构性矛盾

矛盾本质：
  ├── 集约化：芯片选型统一、框架标准化、算子复用、规模降本
  └── 灵活性：业务多样、SLA不同、Oncall可控、新业务快速接入

TA解决方案 = 集约化底座 + 场景化分层：
  ├── Infra层做集约：芯片选型、框架适配、算子优化统一建设
  ├── Serving层做分层：按场景SLA分级(Serving Pool分级)
  └── 应用层接标准化接口：各业务对接Serving API，不感知底层芯片
```

| TA产品 | 解决的问题 | 技术方案 | 优先级 |
|--------|----------|---------|--------|
